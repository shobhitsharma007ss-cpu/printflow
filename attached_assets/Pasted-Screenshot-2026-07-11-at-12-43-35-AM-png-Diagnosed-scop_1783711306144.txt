before i build with replit this is his plan okay? #18 - Authentication & Roles
What & Why
PrintFlow currently has no login — anyone reaching the app sees every screen. Add email + password authentication with three roles (owner, supervisor, operator) so the factory owner controls who can see costing/reports/settings, supervisors manage jobs and inventory, and shop-floor operators only reach the station screens. This is the foundation for safe multi-user use on the plant floor.

Done looks like
Logged-out users see only a login page; every other screen redirects to login.
The seeded owner account can log in and reach all screens.
An operator account logs in and lands directly on the station picker (/floor/stations), and cannot open owner/supervisor screens.
A supervisor account can open Jobs and Inventory but not Costing, Reports, or Settings.
Role gates:
Owner only: Settings, Costing, Reports.
Owner + Supervisor: Jobs, Inventory.
All roles: Floor Monitor, Operator Mode (stations), Job Card.
Logging out returns the user to the login page and clears the session.
Sessions persist across page refreshes.
The sidebar only shows links the current role is allowed to open.
Out of scope
Self-service signup, password reset, and email verification (owner creates users later; only the seeded owner exists now).
Per-user audit trails and activity logging.
OAuth / SSO / third-party login.
A user-management admin UI (adding/editing users) — seed one owner for now.
Steps
Auth backend — Add server-side email/password auth backed by Postgres-stored sessions with hashed passwords. Endpoints for login, logout, and fetching the current user. On failed login return a clear error; never leak whether the email exists.
Users table + seed migration — Add a users table (id, name, email unique, password hash, role, created_at) as a new idempotent migration that creates the table only if missing, and seed a single owner account (email owner@printflow.in, password printflow123, name "Owner"). Also provision the session storage table.
Login page — Build a login screen that visually matches the existing app (reuse the same Card/Input/Button components and styling seen in the costing page). Show inline errors on bad credentials and a loading state while submitting.
Auth context + route guard — Add a client-side auth provider that loads the current user on startup, gates the whole app so logged-out users only see login, and redirects by role. Operator logins land on /floor/stations.
Role gating — Enforce the role matrix on both the routes and the sidebar so disallowed screens are neither navigable nor linked. Wire logout into the existing UI.
Verification — Confirm and report: (1) a logged-out user hitting /jobs is redirected to login, (2) owner login works with the seeded credentials, (3) operator login lands on /floor/stations and cannot reach owner screens, (4) all existing pages still render after login.
Critical constraints
Do NOT regenerate the API client/types wholesale. The generated packages in lib/api-zod and lib/api-client-react are hand-maintained with drift beyond lib/api-spec/openapi.yaml; a full codegen run deletes drifted endpoints and breaks the app. Extend types surgically instead.
Add the new database work as a new numbered migration using IF NOT EXISTS, following the existing idempotent pattern; do not alter or renumber existing migrations.
Reuse the existing session/cookie setup conventions already present; keep session secret in an environment variable, not hardcoded.
Preserve the two existing route trees (operator/job-card full-screen tree and the main AppLayout tree); both must sit behind auth.
